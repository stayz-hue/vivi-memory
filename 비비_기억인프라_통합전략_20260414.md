## 1. 결정 요약

비비의 2층(기억) 인프라를 **5개 독립 도구 조합**으로 구축한다.
각 도구는 있는 그대로 사용하며, 재개발하지 않는다.

| 도구 | 역할 | 배포 방식 | 비고 |
|------|------|----------|------|
| **Honcho** | 대화 기반 추론 메모리 | Hermes Agent 내장, 활성화만 | peer 모델로 고객/담당자 프로파일 자동 축적 |
| **pgvector** | 범용 벡터 검색 | PostgreSQL 확장 설치 | 기존 vividb에 확장만 추가, 별도 서비스 아님 |
| **Qdrant** | 전용 벡터 검색 엔진 | Docker (on-disk 모드) | 판례/약관/케이스 대규모 임베딩 검색 |
| **Zep** | 시간축 메모리 | Docker | 고객별 보험 이력 시간선 추론, 사실의 유효기간 추적 |
| **Neo4j** | 지식 그래프 | Docker | 판례-법령-상해-등급 관계 추론, 판례 10만건 즉시 투입 |

---

## 2. 왜 이 조합인가

### 각 도구가 풀어야 하는 문제가 다르다

- **Honcho**: "이 고객이 어떤 사람인지" — 대화에서 성향/감정/패턴을 자동 추론. Hermes Agent에 이미 내장되어 있어 활성화만 하면 됨. dreaming 기능으로 비비가 쉬는 시간에 고객 이해도 자동 최적화.
- **pgvector**: "이 문장과 비슷한 게 뭐 있는지" — PostgreSQL 안에서 바로 벡터 검색. 별도 서비스 없이 기존 DB에 확장만 추가. 간단한 유사도 검색용.
- **Qdrant**: "10만건 판례 중에서 이 케이스와 유사한 3건" — 대규모 벡터 전용 엔진. on-disk 모드로 RAM 150MB 수준. pgvector로는 10만건 이상에서 성능 한계.
- **Zep**: "김 고객이 2024년 3월에 보험 갱신 → 2025년 1월 해지 → 2025년 6월 사고" — 사실에 시간 태그를 붙여서 시간선 위에 배치. 담당자 교체 감지, 계약 변경 이력 추적. 벡터 검색으로는 불가능한 시간 추론 전담.
- **Neo4j**: "경추 4-5 추간판탈출증 → 장해등급 → 보상 규정 → 유사 판례" — 엔티티 간 관계를 그래프로 연결. 판례 10만건이 4/15 도착하므로 그릇을 먼저 준비.

### pgvector와 Qdrant 둘 다 필요한 이유

pgvector는 PostgreSQL 안에 사는 확장이라 JOIN이 됨 — "진행 중인 케이스 중 유사한 것"처럼 구조화 데이터와 벡터를 동시에 쿼리할 때 강점. 하지만 10만건 이상 전용 벡터 검색은 Qdrant가 훨씬 빠르고 메모리 효율적 (on-disk 모드: 1M 벡터에 135MB). 역할이 겹치지 않음.

## 3. 기각된 대안: PostgreSQL 하나로 통합

### 검토 내용
pgvector + pg_graphql + 커스텀 시간축 로직으로 PostgreSQL 하나에서 벡터 검색 + 그래프 + 시간 추론을 전부 구현하는 안.

### 기각 사유

1. **Supabase 삽질 반복** — Supabase에서 "하나로 다 되니까"로 시작해서 결국 한계 부딪혀 PostgreSQL 로컬로 이관함 (4/13 작업). "하나로 통합"의 유혹이 주는 삽질 패턴을 이미 경험.

2. **삽질방지헌법 1조: 기존 도구에 있으면 만들지 않는다** — Zep의 시간축 추론, Neo4j의 그래프 쿼리를 PostgreSQL에서 커스텀으로 구현하면 "있는 걸 다시 만드는 것"이 됨. 직접 만들면 버그, 기성은 유지보수 0 (삽질방지헌법 36조).

3. **도메인이 다르면 구조도 달라야 한다** (삽질방지헌법 16조) — 범용 DB에서 전문 기능을 흉내내면 어중간한 결과물이 됨. 각 도구가 자기 영역에서 최적화된 걸 그대로 쓰는 게 맞음.

## 4. 작업 순서 결정 배경

### 기억 인프라를 층별 작업(1층 이해 복원 등)보다 먼저 깔기로 함

**이유: 기억은 인프라(배관)이므로 벽(1층) 세우기 전에 깔아야 나중에 안 뜯음.**

- 1층(이해) 복원을 먼저 하면, 이해 결과를 저장할 메모리가 없어서 결국 기억 인프라 깔 때 1층 코드도 다시 고쳐야 함
- 판례 10만건이 4/15 도착 → 그릇(Neo4j + Qdrant)이 먼저 준비되어야 바로 투입 가능
- 크론 엔진 = 능동적 감지의 시간 기반 쌍둥이 (0층 이벤트 감지의 보완)

## 5. 레이어 모델과의 매핑

| Vivi 레이어 | 사용 도구 | 용도 |
|------------|----------|------|
| 0층 감지 | — | 기억 인프라와 직접 관련 없음 |
| 1층 이해 | Honcho | 메시지에서 고객 성향/긴급도 추론 |
| 2층 기억 | **전부** | 이 층의 핵심 인프라 |
| 3층 판단 | pgvector + Qdrant + Neo4j | 유사 케이스 검색 + 판례 추론 → 판단 근거 제공 |
| 4층 행동 | Zep | 시간 맥락 기반 리마인드 ("3개월 전 약속 소환") |
| 5층 학습 | Honcho dreaming | 비비 쉬는 시간에 고객 이해도 자동 최적화 |

## 6. 예상 리소스

| 도구 | RAM (예상) | 디스크 | 비고 |
|------|-----------|--------|------|
| Honcho | Hermes에 포함 | PostgreSQL 공유 | 활성화만 |
| pgvector | PostgreSQL에 포함 | 기존 vividb | 확장 설치만 |
| Qdrant | ~150MB (on-disk) | SSD 의존 | Docker |
| Zep | ~500MB-1GB | Neo4j 공유 가능 | Docker |
| Neo4j | ~500MB-2GB | 판례 건수 비례 | Docker |
| **합계 추가분** | **~1.2-3.2GB** | | 24GB VPS에서 여유 |

현재 VPS 사용량 ~2.6GB + 추가 ~3GB = ~5.6GB. 24GB 중 18GB+ 여유.

## 7. 에르메스 도구 엔진 확장 (관련 결정)

기억 인프라와 함께, Hermes Agent의 도구 확장 방향도 확정:

- **플러그인**: 기존 스킬 시스템 활용
- **MCP**: Claude Agent SDK 연동 (Tool Search로 수천 개 도구 동적 검색)
- **자동 스킬 생성**: Voyager 패턴으로 호프만 계산, 과실비율 조회, 판례 검색 등 손사 특화 도구 무한 추가 가능

## 9. 주의사항

- **신규 버킷 생성 금지** — MinIO 4개 고정 (기존 규칙)
- **Docker 메모리 제한 설정 필수** — Neo4j, Zep에 각각 상한 걸어야 함
- **Honcho AGPL-3.0 라이선스** — 상용화 시 주의 (관리형 서비스 사용 또는 라이선스 확인 필요)
- **판례 임베딩 모델 선택 미정** — 한국어 임베딩 모델 비교 필요 (딥리서치 대상)
- **각 도구 간 데이터 흐름 설계 미정** — 구현 시 정의 필요
