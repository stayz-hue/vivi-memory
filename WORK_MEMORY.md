#### 결정
- BB-OS 보안 원칙: 제로노리지 클라이언트측 암호화 + Shamir's Secret Sharing 키 분할 + 분산/불변 저장소 3축 결합
- 백업은 원본 VPS 보안과 동시 설계 (LUKS 디스크 암호화, SSH 키 전용, fail2ban)
- 암호화 도구: restic (AES-256, 10년차 검증, 오픈소스) — pg_dump 결과물을 평문 업로드 절대 금지
- 저장소 후보 2파: (1) Storj - 분산 P2P, 실질적으로 영장 대상 없음 (2) B2 + Object Lock - WORM 불변성
- 키 관리: Shamir 5-of-3. 5개 조각 보관처 설계는 대표님 개인 신뢰망 영역 (금고/변호사/가족 등)

- 같은 VPS 회사(Contabo Storage)에 백업 — 본사 장애 시 원본·백업 동시 사망

#### 트레이드오프 (대표님이 반드시 이해해야 할 2가지)
1. "완벽한 보안"의 대가 = 키 잃으면 영구 소실. Shamir로 완화 가능하나 조각 보관처 5곳 설계 필수
2. "영장 포렌식 불가"가 비비 서비스화 시 양날의 검. 범죄 악용 시 서비스 제공자(대표님) 형사 책임 가능성. 개인 용도는 무관

#### 단계별 로드맵
- Phase 1 (이번 주): restic + Storj(또는 B2 Object Lock) + Shamir 3-of-5 구축
- Phase 2 (다음 주): 복구 시뮬레이션 필수 (다른 머신에서 조각 3개 모아 실제 복원 검증)
- Phase 3 (지속): 키 조각 보관처 순회 점검, 저장소 다중화

#### 인사이트
- "비트코인급 보안" 레토릭을 해체하면 4개 요구사항: 제로노리지 / 사용자 독점 키 / 키 분실 방지 / 강제 탈취 방지. 검증된 오픈소스 조합(restic+Storj+Shamir)이 머스크 XChat보다 실질 더 강함
- 백업 하나만 보안해도 의미 없음. 원본 VPS, 전송로, 접근제어, 내부자 위험을 동시에 봐야 BB-OS 보안 첫단추 완성
- Storj가 "진짜 peer-to-peer" 모델. 데이터 조각이 전세계 수백 노드에 분산되어 한 노드 영장 받아도 쓸모없음. 머스크가 말한 모델을 실제로 구현한 것

#### 결정 (앞선 드래프트 일부 수정)
- 비비 보안 설계의 근본 원칙 변경: 단순 "사용자가 키 보유" → "본인도 즉시 접근 불가" 모델로 재설계
- 공개키 암호 패러다임 채택: 평상시 VPS/Agent는 공개키로 암호화 저장만, 복호화 개인키는 분산 보관되어 본인도 조립 절차 거쳐야 접근
- 기술 조합 재선정: restic+Storj → age + BIP39/SLIP-39 + Shamir 3-of-5 + 분산 저장소
- "범죄자 악용 우려" 프레임 폐기: 비비는 메시저가 아닌 자동 수집+회상 시스템. 도구 성격 자체가 범죄자 동기와 충돌. 법 지키는 시민의 인지 증폭 도구로 포지셔닝 확정

#### 폐기한 오답
- "사용자가 키 가지면 제로노리지 달성"으로 단순화한 것 — 사용자가 공격 단일지점이 됨. 영장·강요·사회공학 시 원스톱 함락
- restic 단독 추천 — restic은 "평상시 본인 접근 가능" 전제라 "본인도 즉시 접근 불가" 모델과 구조적 불일치
- Signal/ProtonMail 식 "범죄 악용 리스크"를 비비에 기계적으로 적용한 것 — 비비는 메시저가 아니라 성격이 다름

#### 핵심 원칙 (새로 확정)
1. 본인도 즉시 접근 불가가 기본. 열람은 Shamir 조각 재조립 절차 필수
2. 아키텍처의 "의도적 지연"을 문서화해 법적 방어 근거 확보 (증거인멸죄 경계)
3. 암호 알고리즘은 검증된 오픈소스, 통합 레이어는 비비 자체 개발 (Schneier 원칙)
4. "비트코인급 보안"의 실체 = 비트코인 지갑 생태계 10년 노하우 (BIP39 시드 + 분산 보관)

#### 인사이트
- 대표님 지적 핵심: "나도 접근 제한적이어야 논리 구조 맞음" — 제로노리지는 서비스 제공자뿐 아니라 사용자 본인에 대해서도 적용되어야 완성
- Time-lock encryption(tlock/drand)은 "영장 즉시성 파괴" 관점에서 흥미로운 신규 기술. 단 성숙도 검증 구간
- 비비의 범죄자 배제는 기술적 차단이 아니라 "도구 성격 자체가 선별자 역할" — 자동 수집은 증거 생산 기계라 범죄자는 동기가 없음. 기록하지 않는 게 범죄자의 본능

#### 딥리서치 후보 (우선순위 미정)
1. "일상 자동 수집 시스템"(Rewind.ai, MS Recall, Limitless 등) 보안 설계 비교 + 사고 사례
2. "본인도 즉시 접근 불가" 아키텍처 프로덕션 사례 (법조계, 언론인, 활동가, 은행)
3. age + BIP39/SLIP-39 + Time-lock 조합 성숙도 검증
4. 한국법상 신탁·분산 키 보관의 법적 유효성 (영장 대응, 증거인멸죄 경계)

#### 보류
- 저장소 선택 (Storj vs B2+Object Lock) — 딥리서치 후 재결정
- Shamir 조각 보관처 5곳 선정 — 법적 유효성 확인 후

#### 결정 (패러다임 전환)
- BB-OS 보안 모델: "수학적 완벽성" → "입 무거운 집사" 모델로 전환
- 비비는 "대표님 기억의 외부화 장치"이며, 보호 대상은 "기록 자체"가 아니라 "기록이 밖으로 나가는 통로"
- 철학적 근거: "나만 아는 비밀은 나만 얘기 안 하면 된다" - 인류가 집사·비서·변호사에게 비밀 맡겨온 구조의 디지털 재현
- "완전한 보안 불가능" 현실 인정, "관리 가능한 리스크" 모델로 전환

#### 폐기한 접근
- 공개키 암호 + Shamir + Time-lock 조합으로 "본인도 즉시 접근 불가" 구현 - 비비가 상시 연산해야 하므로 어차피 평문 필요. 암호화 복잡도만 폭발
- "수학적으로 완벽한 영장 방어" 추구 - 비비 특성상(기억·학습·회상) 실현 불가
- FHE(동형암호) 같은 이론적 해결책 - 실용 수준 아님

#### 집사 모델의 4층 구조
1. 저장: 평문 OK (연산 필요), LUKS 디스크 암호화로 물리 탈취 방어
2. 외부 노출: 텔레그램 봇 단일 채널, 외부 포트 최소화, SSH 대표님 키만
3. 인증: 다요소 (텔레그램 계정 + 행동 지문 + 필요시 음성). 3요소 중 2이상 매칭
4. 발설 통제: 비비는 대표님 외 응답 안 함. 외부 API로 원본 최소 전송. 로컬 LLM 우선

#### 대표님 3가지 제안의 흡수
- 1번 "나만 안 얘기하면 됨" = 집사 모델의 철학적 기반
- 2번 "위기 감지 전면 차단" = Dead man's switch + Panic command + 이상 탐지
- 3번 "지문 인증" = Behavioral biometrics (타이핑·음성·위치 다요소). 수신/저장은 자동, 조회/대화만 인증

#### 위협 모델 재정의
- 일반 해커/해킹 시도: 집사 모델로 충분 방어
- 물리 탈취: LUKS로 완화
- 내부자 없음 (1인 운영)
- 사법당국/영장: 법 영역, 기술로 못 막음. 법적 전략으로만 대응
- 국가급 공격자: 방어 포기 (개인 도구 스코프 밖)

#### 딥리서치 주제 재편 영향
- 주제 3 (암호학 기술 성숙도) → 불필요화
- 주제 2 (분산 키 보관) → 축소. dead-man/panic 부분만 관심
- 주제 4 (한국법) → 여전히 필수
- 주제 1 (유사 제품 사고) → 핵심으로 승격. Rewind/Recall이 집사 모델에서 어디서 실패했는지 배울 가치 높음

#### 인사이트
- 기술 완벽주의의 함정: 복잡도 증가 → 대표님 자신도 못 씀 → 결국 비비 사용 포기 → 보안의 본래 목적 상실
- 비비의 특수성(1인 운영, 텔레그램 단일 외부 접점)이 오히려 보안에 유리. "방 하나짜리 집"이라 잘 숨기기 쉬움
- 사회적 신뢰 구조의 디지털 재현이 수학적 완벽성보다 현실적이고 운영 가능

#### 결정 (또 한 번의 패러다임 재정렬)
- BB-OS 인증 모델: "생체 인증(단발 지문·얼굴·음성)" 폐기 → "체화 기반 연속 인증" 방향 확정
- 핵심: 사용자의 오랜 상호작용으로 쌓인 고유 패턴(문체·지식·판단·기억)이 키 생성의 입력값이 되는 구조
- 학술 영역: Fuzzy Extractor / Biometric Cryptosystem
- 시간이 지날수록 강해지는 인증 (일반 보안과 반대 방향)

#### Claude 실수 인정 및 교훈

  2차: "체화 데이터 인증" → "지문/얼굴/음성 단발 인증" 축소 → 대표님이 "단발 아니야" 교정

#### 대표님 그림 정확히
- 저장매체: 국가권력급 공격자도 암호학적으로 못 풀 수준 (현실적으로는 "경제적 방어"로 재정의 - 공격 비용 > 정보 가치)
- 접근 방법: 사용자의 체화된 인지 패턴 자체가 키
- 인증: 단발 아닌 연속, 누적, 오랜 시간으로 강화됨
- 계승: 유서 아닌 도제식 전수. B가 A의 체화를 공유하는 긴 과정

#### 비비 철학과의 일관성
- 비비 5가지 비밀 1번(전문적 직관의 외부화), 5번(한 사람의 인지 보존)과 같은 기둥
- 보안 구조 = 비비 철학의 자연스러운 연장선. 따로 노는 게 아님

#### 기술 가능성 정직한 평가
- 가능: 연속 행동 인증(은행 실사용), 지식 챌린지(비비-대표님 공유 기억), AES-256+하드웨어 보안 칩 조합
- 어려움: 완전한 Fuzzy Extractor는 학술 단계, 프로덕션 라이브러리 부족
- 불가: "절대 못 뚫는" 시스템은 존재 안 함. "뚫을 가치 없게"로 재정의

#### 딥리서치 주제 재편 (3차)
1. Fuzzy Extractor / Biometric Cryptosystem 학술·프로덕션 성숙도
2. 연속 인증(Continuous Authentication) 실무 적용 사례
3. 체화된 인지를 암호학적으로 다루는 LLM 시대 최신 연구

- 체화 인증은 "문 앞에서 누구 들어올지" 통제. 서버 자체 뚫리는 시나리오는 별도 축 필수
- 장기 방향: 사무실 자체 서버로 이전 (VPS 제공자 통제권 제거)

- Contabo 직원 하이퍼바이저 메모리 덤프: 체화 인증 못 막음
- 정부 영장으로 VPS 회사 요청: 체화 인증 못 막음, Contabo가 디스크+메모리 덤프 넘김
- 구조적 취약 윈도우: 대표님이 비비 쓰는 시간 = 메모리에 평문 존재

#### 결정 (보안 패러다임 5번째 전환)

- 기술 번역: Distributed Fragmented Storage + Threshold Reconstruction
- 단일 서버 + "재량권 없는 관리자"(B안)보다 한 단계 위 구조
- Claude 실수: A안을 즉시 "불가능"으로 축소 번역. Reed-Solomon + 분산 파편화 가능성 제시 안 함

#### 기술 실체

- 체화 인증이 재구성 트리거 역할
- 사용 후 즉시 파기 → 다시 "의미는 존재하지 않는 상태"
- 관리자는 물리적으로 존재하지만 "조각 관리자"일 뿐, 전체는 어디에도 집중되지 않음

#### 진짜 달성 여부

- "완전 Zero-Knowledge" - 개인정보보호법 제26조 감독의무, AI 기본법 제34조와 충돌

#### 구조 분리 설계 (해법)
1. 비비 = SaaS 제공자 (Zero-Knowledge): 고객 데이터 접근 불가, 영장 시 암호문만 제공
2. 각 고객 손사 = 본인 Passkey로 자기 데이터 암호화, 본인 책임
3. 대표님 본인 = 비비의 한 고객으로 분리, 본인 업무는 본인 책임
4. 고객 영장은 비비 아닌 해당 고객에게 직접 가야 함

#### 프로톤메일 모델 (한국법 호환)
- 본문: E2E (제공자 못 봄)
- 메타데이터: 제공자 보유 (개인정보법 제29조 접속기록 보관 + 제26조 수탁자 감독의무 충족)
- 이 분리가 비비의 현실적 최대 방어선

#### 비비 공식 약속 문구 (초안)
"비비는 고객 데이터를 기술적으로 볼 수 없습니다. 영장이 와도 우리가 제공할 수 있는 것은 암호문뿐입니다. 고객 키 없이는 비비 개발자조차 복호화할 수 없습니다. 다만 접속 메타데이터는 법적 의무에 따라 일정 기간 보관되며, 이는 규제 감독을 위함입니다."

#### 남는 3개 구멍 (정직 공개용)
1. 메타데이터 (접속 기록, IP, 용량) - 법적 보관 의무
2. AI 학습·처리 순간 평문 노출 - TEE(Trusted Execution Environment)로 최소화
3. 대표님 본인 데이터 - 대표님 자신 책임 영역

#### 비비의 진짜 차별화 (오늘 도달)

#### 확정 모델: Obsidian 모델

- 비비 회사가 갖지 않는 것: 사용자 원본 데이터, 키, 서버 접근권
- 데이터 저장: 사용자 본인 하드웨어 (VPS 또는 집/사무실 서버)
- 영장 시 비비 회사: "줄 것 없음" - 사실. 영장은 각 사용자에게 개별

- 비비 회사 ≠ 사용자 데이터 저장소

- customer_sensitive 테이블은 만들되 당장은 사용 안 함. C1-2-2 or 별도 작업에서 GAS v3 편집해 포함
- cache_key 기반 중복 방지 (같은 조회 재실행 시 덮어쓰기)

#### 외부 도구 검토: OpenCrab (AlexAI-MCP)
- 정체: MCP 서버 형태의 MetaOntology OS. Neo4j(그래프) + ChromaDB(벡터) + MongoDB(문서) + PostgreSQL(정책/ReBAC) 4-store 통합
- space 구조: Subject / Resource / Concept / Outcome / Evidence / Policy
- 핵심 기능: query_ontology(하이브리드 검색), manage_entity, impact_analysis

#### 비비 도입 결정: ❌ 도입 안 함
- 기능 중복: 비비는 이미 Neo4j + Qdrant + Graphiti + Honcho + MinIO + PostgreSQL로 더 세분화 돼있음. OpenCrab의 ChromaDB 단일 벡터 역할을 비비는 3개(Qdrant/Graphiti/Honcho)로 분리
- 방향 역행: OpenCrab=범용 OS, 비비=손해사정 도메인 특화. 범용 OS 얹으면 참조본 v2 특화 설계(고객→사고→청구, 6단계, 12유형)가 OpenCrab space 스키마에 왜곡될 위험
- 삽질 방지 헌법 1조(기존 도구에 있으면 만들지 않는다) + 16조(외부 오픈소스 구조 검증 없이 이식 금지) 위반

#### 참고 각도 1개 — 나중에 Phase 3 이후 검토
- "impact_analysis" 발상 = 역방향 질의 구조
- 예: "이 담당자 바뀌면 진행중 어느 케이스에 영향?", "장해 11급 나오면 어느 보험사에서 재청구 가능?"
- Neo4j에 이미 있는 관계 그래프 활용도를 더 뽑는 방식
- 현재 우선순위 아님 — C1-2 완료 → Phase 2A/2B → Phase 3 판례 끝난 뒤 재검토

#### 원칙 재확인
- 외부 온톨로지 도구 유혹에 흔들리지 않음. 비비 참조본 v2 자체가 이미 손해사정 도메인 온톨로지
- "모으는 건 범용, 쓰는 건 특화" — 비비는 쓰는 쪽

#### 외부 도구 검토: Fabric (danielmiessler, GitHub 40k+ stars)
- 정체: Go 기반 오픈소스 AI 프롬프트 프레임워크. 프롬프트를 "패턴"으로 모듈화 → 조립식 사용
- 성격: 커뮤니티 패턴 공유/크라우드소싱 인프라
- 비비 도입: ❌ — 비비는 대표님 1인 전용, Fabric은 집단 지성 인프라. 방향 반대

#### 진짜 인사이트: 인지 오프로딩 vs 패턴 해체 (비비 설계 철학 직결)
- 스레드 핵심 명제: "패턴을 복사하는 사람에겐 퇴화, 패턴을 해체하는 사람에겐 성장. 도구가 사고를 높이는지 낮추는지는 사용자가 결정한다."
- 비비에 적용: 비비 = 대표님 전문적 직관의 외부화(5가지 비밀 1번). 잘못 설계되면 외부화가 아니라 **퇴화** 유발

#### 비비가 대표님을 퇴화시킬 수 있는 장면들
1. 장해등급 제안 → 근거 없이 승인 → 왜 11급인지 감각 소실
2. 보험사 담당자 패턴 자동 적용 → 직접 업데이트 안 함 → 감각 고정
3. 카톡 답변 초안 무수정 전송 → 고객별 톤 감각 소실

#### 반대 방향(패턴 해체) 설계 원칙
- 판단 결과만이 아니라 근거 체인 노출 (Neo4j 추론 경로 가시화)
- 자동 전환 발생 시 "어떤 신호로 전환했는지" 1줄 설명
- 반복 업무 발견 시 "예외 N번" 함께 제시 → 예외에서 감각 업데이트
- 비비 제안 기각률 추적 → 기각 많은 영역 = 비비 틀린 영역 = 대표님 감각 우위 영역 식별

#### 비비 헌법 추가 조항 (신설 필요)
> **비비는 결과만 주는 게 아니라 근거를 같이 준다. 대표님이 근거를 볼지 말지는 선택이지만, 근거 자체가 없으면 그건 비비가 대표님을 퇴화시키는 것이다.**

#### 연결되는 기존 메모리

- 종결 이벤트: 보험사별 합의 (보험사 단위)
- 수수료 정산 이벤트: 여러 보험사 묶어서 한 번 정산 (보험사 묶음 단위)
- 두 이벤트가 늘 동시는 아님 (월마감 일부종결 수수료 선수 케이스 존재)
- 따라서 분리 설계 필요

#### 신규 테이블 2개

**insurer_outcomes (보험사별 종결 기록, case 하위)**
- id, case_id, insurer_name, status, settlement_rate_percent, settlement_condition, compensation_amount, settled_at, created_at
- 각 보험사마다 합의율(예: 삼성 2%, KB 3%) 개별 저장
- 총평과 Honcho 기억의 단위 = 이 레코드

**fee_settlements (수수료 정산 이벤트)**
- id, case_id, insurer_outcome_ids(배열), gross_compensation, fee_rate_percent, fee_amount_with_vat, settled_at, notes
- 월마감 일부종결 선수도 자연스럽게 row 하나로 표현
- case에 여러 fee_settlements row 가능

#### 명령어 형식
기본: `{고객명} {보험사들} 종결 수령액 {X}만원 수수료 {Y}%(부가세포함) {Z}만원`
예: 김철수 삼성화재 KB손해 메리츠화재 종결 수령액 1000만원 수수료 15%(부가세포함) 165만원

월마감 선수 엣지케이스: `{고객명} {보험사들} 수수료 선수 수령액 {X}만원 수수료 {Y}%(부가세포함) {Z}만원`
- case.status는 진행중 유지, fee_settlements에만 row 생성

#### case 상태 판정 룰
- 모든 insurer_outcomes가 '종결' → cases.status='종결'
- 아니면 '진행중' 유지
